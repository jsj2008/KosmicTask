4. Notes

Event loops

Python-based applications that use a Carbon/Cocoa event loop can import the osax module as normal, but should not use it before the event loop has started as sending Apple events outwith the main event loop can disrupt the process's normal event handling.

64-bit limitations

The osax module only supports dynamic retrieval of scripting addition terminology when running in 32-bit processes.

The osax module includes static terminology for Mac OS X 10.6's Standard Additions scripting addition, allowing 64-bit Python processes to use Standard Additions as normal. Other scripting additions can be used in 64-bit processes (assuming the additions themselves are 64-bit aware) as long as you supply the OSAX.__init__ constructor with static terminology for that addition.

To do this, first call the appscript.terminology module's dump function, passing it two arguments: the path to the scripting addition, and the path to the 'glue' module to be created. For example:

#!/usr/bin/arch -i386 /System/Library/Frameworks/Python.framework/Versions/2.6/bin/python2.6

from appscript import terminology

terminology.dump('/Library/ScriptingAdditions/Foo.osax', 
        '/Library/Python/2.6/site-packages/fooglue.py')

Once you have created a glue module and placed it on Python's module search path, import it into your script and use it as the terms argument to the OSAX.__init__ constructor, like this: 

import osax, fooglue

foo = osax.OSAX('Foo', terms=fooglue)

GUI interaction

In Python 2.4 and later, the python interpreter will automatically connect to the Window Manager when sending scripting addition commands that require GUI access (e.g. display_dialog). In Python 2.3, sending GUI-related commands will cause a "no user interaction allowed" error unless the pythonw interpreter is used instead.
